# Normalize Internal AI-Generated Fields to a Configured Language

## Context

The AI pipeline (`ArticleAnalysisWorker` + the `Infrastructure/AI/*` services) produces several
fields that should be semantically uniform across the system so editors, publishers, and
downstream consumers see consistent language. Today these fields are inconsistent because:

- Some prompts hardcode "Ukrainian (uk)" in natural language (summary updater, contradiction
  detector, telegram generator, event classifier contradiction descriptions, event title
  generator system prompt, analyzer summary).
- Some prompts do not enforce any language and emit text in the source article's language
  (notably `HaikuKeyFactsExtractor` — its system prompt says nothing about language, so
  `Article.KeyFacts` for a Russian article comes back in Russian, for a Polish article in
  Polish, etc.).
- `AnthropicOptions.OutputLanguage = "uk"` already exists in `Infrastructure/Configuration/AiOptions.cs:30`
  but is **orphaned** — nothing reads it. Changing it in `Worker/appsettings.Development.json`
  has zero effect today.

As a result:

1. `Article.KeyFacts`, which is consumed by `ClaudeEventClassifier.BuildCandidateBlock`,
   `ClaudeContradictionDetector.BuildUserPrompt`, and `ClaudeContentGenerator.BuildArticlesSection`,
   arrives in multiple languages inside the same `CANDIDATE EVENTS` block — this can degrade
   the LLM's matching and contradiction-detection quality.
2. The project's single-language intent (currently Ukrainian) is expressed as a copy-pasted
   string in six different prompt files plus one inline `SystemPrompt` constant. Changing the
   output language requires editing six prompts.
3. User-facing external fields (article `Title`, `OriginalContent`, `Source.Name`) must stay
   in the original source language — this is intentional and must not be broken by the change.

### Internal vs External Fields

Based on the domain models in `Core/DomainModels/`:

**Internal (must be normalized to configured language):**

| Field | Model | Produced by | Current state |
|---|---|---|---|
| `Article.Summary` | `Core/DomainModels/Article.cs:20` | `ClaudeArticleAnalyzer` / `GeminiArticleAnalyzer` via `analyzer.txt` | Already forced to Ukrainian (hardcoded) |
| `Article.KeyFacts` | `Core/DomainModels/Article.cs:21` | `HaikuKeyFactsExtractor` (inline prompt) | **NOT normalized — emitted in article language** |
| `Event.Title` | `Core/DomainModels/Event.cs:6` | `HaikuEventTitleGenerator` (inline prompt) | Already forced to Ukrainian (hardcoded) |
| `Event.Summary` | `Core/DomainModels/Event.cs:7` | `ClaudeEventSummaryUpdater` via `event_summary_updater.txt` | Already forced to Ukrainian (hardcoded) |
| `EventUpdate.FactSummary` | `Core/DomainModels/EventUpdate.cs:12` | `ClaudeEventClassifier.new_facts` via `event_classifier.txt` | **NOT normalized — inherits article language** |
| `Contradiction.Description` | `Core/DomainModels/Contradiction.cs:9` | `ClaudeContradictionDetector` + `ClaudeEventClassifier` contradictions branch | Already forced to Ukrainian (hardcoded) |

**External (must remain in original source language):**

| Field | Model | Reason |
|---|---|---|
| `Article.Title` | `Core/DomainModels/Article.cs:16` | Scraped from source, shown verbatim, used as link anchor |
| `Article.OriginalContent` | `Core/DomainModels/Article.cs:8` | Original source text |
| `Article.OriginalUrl` | `Core/DomainModels/Article.cs:10` | N/A — URL |
| `Article.Tags` / `Article.Category` | `Core/DomainModels/Article.cs:17-18` | Tags are already spec'd "in the same language as the article" in `analyzer.txt:17`; category is an enum-like controlled vocabulary — out of scope |
| `Article.Language` | `Core/DomainModels/Article.cs:28` | Detected language code (metadata) |
| `Source.Name` / any `PublishTarget.Name` | — | External identifiers |
| `Publication.Content` (Telegram post body) | generated by `ClaudeContentGenerator` via `telegram.txt` | This is the final external publication. **Per user intent: this stays in whatever language the publication channel expects** — today that is Ukrainian, hardcoded in `telegram.txt`. See "Scope" below. |

### Scope boundary on Telegram publications

The user's phrasing said "external — on original language". Literally applied, this would mean
Telegram posts should be in each source article's language, which contradicts the current
product behavior (one Ukrainian Telegram channel). The user's real intent is:

- **Internal AI-produced metadata fields** (summary/key facts/event summary/event
  title/event update facts/contradiction descriptions) → normalized to one configured language.
- **Raw scraped article fields** (`Title`, `OriginalContent`, source metadata) → stay in
  original language. These are never rewritten by AI today, so no change is needed.
- **Publication content** (Telegram post body) → controlled by the `telegram.txt` prompt and
  `PublishTarget.SystemPrompt`. **Out of scope for this ADR** — publication language is a
  per-channel concern (a channel decides its own audience language), not a system-wide
  normalization concern. Keep `telegram.txt` as-is.

## Options

### Option 1 — Use `{language}` placeholder substitution in all prompt files

Add a single `NormalizationOptions.TargetLanguage` config key. Before loading each prompt,
replace a `{OUTPUT_LANGUAGE}` placeholder token with the configured value. Update all
`.txt` prompts to use the placeholder instead of the literal word "Ukrainian".

**Pros:**
- Zero code duplication of the language setting.
- Works for every prompt uniformly.
- The prompt files still read naturally for a human reviewer (the token is obvious).

**Cons:**
- Requires changes to `PromptsOptions.ReadPrompt` (it currently just does `File.ReadAllText`).
- The placeholder convention is new to the project — no existing ADR or skill documents it.
- Still leaves the inline prompts in `HaikuKeyFactsExtractor` and `HaikuEventTitleGenerator`
  needing a different mechanism (string interpolation in the class constructor).

### Option 2 — Pass target language as a parameter into each AI service, build prompt dynamically

Inject `IOptions<LanguageNormalizationOptions>` into every AI service that emits normalized
text. Each service builds its final system prompt by interpolating the language into the
template (either a `.txt` file template or an inline string).

**Pros:**
- Consistent with how `HaikuKeyFactsExtractor` already owns its inline prompt.
- Strongly typed — no string-replacement-based templating.
- Per-service flexibility if we ever want different target languages for different fields.

**Cons:**
- Every AI service constructor signature changes; DI registrations in
  `InfrastructureServiceExtensions.AddAiServices` all grow a parameter.
- More boilerplate per class.
- Adds coupling from every AI service to the options class.

### Option 3 — Keep hardcoded Ukrainian in prompts, only fix `KeyFacts` and `EventUpdate.FactSummary`

Minimal scoped fix: do not introduce configuration at all. Just add a Ukrainian-language
instruction to `HaikuKeyFactsExtractor.SystemPrompt` and to the `event_classifier.txt`
`new_facts` instruction. Leave everything else alone. The `OutputLanguage` option stays
orphaned (or is deleted).

**Pros:**
- Smallest possible change set.
- Ships the actual bug fix (key facts language mismatch) immediately.

**Cons:**
- Does not address the user's explicit request to "возможно вынести в конфиги эту настройку"
  (move the setting to configs).
- Leaves the hardcoded-Ukrainian problem in six prompts — switching language later is still a
  six-file edit.
- Leaves `AnthropicOptions.OutputLanguage` orphaned or requires a separate cleanup.

## Decision

**Option 1 — `{OUTPUT_LANGUAGE}` placeholder substitution in prompts, combined with the same
substitution for inline prompts inside the two classes that don't use `.txt` files.**

Rationale:

1. The user explicitly asked to put the setting into config. Option 3 ignores that; Option 2
   meets it but with disproportionate boilerplate for what is, semantically, one shared string.
2. The placeholder approach keeps each prompt file self-contained and human-readable. A reviewer
   reading `event_summary_updater.txt` will see `Write the summary in {OUTPUT_LANGUAGE}` and
   understand immediately.
3. The project already has precedent for prompt-file-based configuration (`PromptsOptions`) and
   for plain-string config values (`AnthropicOptions.OutputLanguage`). Combining the two is a
   natural extension, not a new pattern.
4. Scope: one global target language, single setting, applied to all internal AI output.
   Per-source overrides are explicitly out of scope (see "Non-goals" below).

### Concrete Decision

**Config shape.** Rename and relocate the existing orphan `AnthropicOptions.OutputLanguage`
into a new top-level `AiOptions.Normalization` section (language normalization is model-agnostic
and not an Anthropic concern):

```
"Ai": {
  "Normalization": {
    "TargetLanguage": "uk",
    "TargetLanguageName": "Ukrainian"
  },
  "Gemini": { ... },
  "Anthropic": { ... }  // OutputLanguage removed here
}
```

- `TargetLanguage` — ISO 639-1 code used for `Article.Language`-style metadata.
- `TargetLanguageName` — full language name in English for embedding into prompts (LLMs respond
  better to "Ukrainian" than to "uk" inside natural-language instructions).

Implementation of the class follows `.claude/skills/code-conventions/SKILL.md` Options pattern:
a nested `NormalizationOptions` class inside `AiOptions`, same style as the existing
`GeminiOptions` and `AnthropicOptions` nested classes.

**Prompt placeholder contract.**

- Add a single placeholder token: `{OUTPUT_LANGUAGE}`.
- Placeholder substitution happens in `PromptsOptions.ReadPrompt` (or a new
  `BuildPrompt(string relativePath)` helper) by replacing `{OUTPUT_LANGUAGE}` with
  `AiOptions.Normalization.TargetLanguageName`. `PromptsOptions` will need access to the
  normalization value — inject it through DI rather than keeping the file-reading static helper.
- If a prompt contains no placeholder, substitution is a no-op (e.g. `telegram.txt` stays
  unchanged).

**Prompt file edits** (replace literal "Ukrainian" / "Ukrainian (uk)" with placeholder):

| File | Line(s) | Change |
|---|---|---|
| `Infrastructure/AI/Prompts/analyzer.txt` | 19 | "in Ukrainian (uk)" → "in {OUTPUT_LANGUAGE}" |
| `Infrastructure/AI/Prompts/event_summary_updater.txt` | 5 | "in Ukrainian (uk)" → "in {OUTPUT_LANGUAGE}" |
| `Infrastructure/AI/Prompts/event_classifier.txt` | 7 | "in Ukrainian" → "in {OUTPUT_LANGUAGE}" |
| `Infrastructure/AI/Prompts/event_classifier.txt` | — (new rule) | Add: "Write all `new_facts` values in {OUTPUT_LANGUAGE}, regardless of the article language." |
| `Infrastructure/AI/Prompts/contradiction_detector.txt` | 5, 33 | "in Ukrainian" → "in {OUTPUT_LANGUAGE}" |
| `Infrastructure/AI/Prompts/telegram.txt` | — | **No change.** Publication language is per-channel (out of scope). |

**Inline prompt edits** (these two classes don't use `.txt` files):

- `Infrastructure/AI/HaikuEventTitleGenerator.cs:14-17` — inject `IOptions<AiOptions>`,
  interpolate `Normalization.TargetLanguageName` into the system prompt where "Ukrainian" is
  currently hardcoded. Update DI registration in
  `Infrastructure/Extensions/InfrastructureServiceExtensions.cs:122-126` accordingly (pass the
  language name into the constructor).
- `Infrastructure/AI/HaikuKeyFactsExtractor.cs:24-27` — **this is the primary bug fix.** Add an
  explicit language instruction:
  `"Respond in {TargetLanguageName}, regardless of the input language."` Inject the language
  via the constructor the same way.

**EventClassificationResult.NewFacts pipeline.** The `new_facts` array returned by the
classifier (consumed in `ArticleAnalysisWorker.TrySaveEventUpdateAsync` and stored into
`EventUpdate.FactSummary`) is currently not language-constrained. After this change, the
prompt edit on `event_classifier.txt` handles it — no worker code change required.

### Non-goals (explicitly out of scope)

- **Per-source target language.** Keep it global; single `TargetLanguage` value. A future ADR
  can extend `Source` with an override if needed.
- **Backfill / re-normalization of historical `Article.KeyFacts`, `EventUpdate.FactSummary`,
  `Event.Summary`, `Event.Title`, or `Contradiction.Description` rows.** Only new pipeline
  runs produce normalized output. Existing rows stay in whatever language they already have.
  If a full re-analysis is desired later, it's a separate migration task (re-queue articles with
  `Status = Pending`).
- **Publication content language** (`Publication.Content`, `telegram.txt`). Per-channel concern.
- **UI locale / i18n** (`UI/`). Unrelated to AI-produced domain fields.
- **Validation/enforcement** that the LLM actually obeyed the instruction. We trust the model
  output as we do today; no post-hoc language detector.

## Follow-up: Unify prompt storage strategy on text files

While implementing this ADR, the project was found to have two inconsistent strategies for
storing AI system prompts:

- **Text files** under `Infrastructure/AI/Prompts/*.txt`, consumed through `PromptsOptions`
  (used by 6 services: `ClaudeArticleAnalyzer`, `GeminiArticleAnalyzer`,
  `ClaudeEventSummaryUpdater`, `ClaudeEventClassifier`, `ClaudeContradictionDetector`,
  `ClaudeContentGenerator`).
- **Inline strings** inside the class (`HaikuKeyFactsExtractor`, `HaikuEventTitleGenerator`).

This inconsistency forced the `{OUTPUT_LANGUAGE}` normalization to be implemented twice:
centrally inside `PromptsOptions.ReadPrompt` for file-based prompts, and per-class via a
constructor-injected `targetLanguageName` + manual string interpolation for the two inline
ones. Every future cross-cutting prompt operation (templating, versioning, per-env overrides,
prompt review) would pay the same tax.

### Decision

Unify on text files. Move both Haiku inline prompts to:

- `Infrastructure/AI/Prompts/haiku_key_facts.txt`
- `Infrastructure/AI/Prompts/haiku_event_title.txt`

Both use the `{OUTPUT_LANGUAGE}` placeholder and are read through `PromptsOptions` like every
other prompt. The two Haiku class constructors stop taking `string targetLanguageName`;
instead they take the already-substituted system prompt string (the pattern used by every
other AI service in `AddAiServices`).

### Rationale

- **One mechanism for placeholder substitution** — `PromptsOptions.ReadPrompt` handles it
  uniformly; no special cases per class.
- **Prompt editing without recompilation** — consistent with how the other 6 services
  already work. Prompt tweaks are data edits, not code edits.
- **Simpler DI** — `InfrastructureServiceExtensions.AddAiServices` no longer passes
  `targetLanguageName` into constructors; all prompts flow through `PromptsOptions`.
- **Discoverability** — all prompts live in one folder, reviewable together.

### Scope

- Add `HaikuKeyFacts` and `HaikuEventTitle` properties to `PromptsOptions` (with their
  path constants).
- Move the inline prompt bodies into the two new `.txt` files, replacing literal language
  references with `{OUTPUT_LANGUAGE}`.
- Change constructor signatures of `HaikuKeyFactsExtractor` and `HaikuEventTitleGenerator`
  to accept `string systemPrompt` in place of `string targetLanguageName`.
- Update DI registrations in `InfrastructureServiceExtensions.AddAiServices`.
- Update the existing language-injection tests to assert on the substituted prompt string
  passed to the constructor rather than on a private `_targetLanguageName` / `_systemPrompt`
  field via reflection.

### Non-goals (for the follow-up)

- No change to `AiOptions` or `NormalizationOptions`.
- No change to `appsettings` files.
- No change to the `{OUTPUT_LANGUAGE}` placeholder contract.
- No change to the other 6 AI services.
