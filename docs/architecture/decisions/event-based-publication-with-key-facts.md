# Event-Based Publication with Key Facts Extraction

## Status

Proposed

## Context

The current publication pipeline is article-centric: the editor approves individual Articles, which creates Publication entities tied to a single Article. The `IContentGenerator.GenerateForPlatformAsync` receives one Article and generates content from that article's title, content, category, tags, and sentiment.

This creates two problems:

1. **No key facts extraction.** Articles have a `Summary` field (generated during analysis) but no structured representation of the important factual claims within the article. The summary is a prose paragraph, not a list of discrete facts that can be compared, merged, or used for structured publication generation.

2. **Publication is article-scoped, not event-scoped.** An Event groups multiple Articles about the same story, but the publication still generates content from a single article. When an Event has 5 articles from different sources, the publication only uses one article's perspective. The Event.Summary exists but is not used as the primary input for content generation.

The desired behavior:
- Each Article gains a `KeyFacts` field (`List<string>`) generated by a cheap AI model (Haiku) during the analysis pipeline.
- The editor approves Events, not individual Articles.
- When an Event is approved, the publication content is generated from the Event.Summary plus the KeyFacts and Summaries of all Articles belonging to that Event, producing a richer, multi-source publication.

### Relationship to Existing ADR

The `worker-pipeline-refactoring.md` ADR already consolidates workers to 3 (SourceFetcherWorker, ArticleAnalysisWorker, PublicationWorker) and establishes that:
- Article analysis happens in ArticleAnalysisWorker (enrichment + embedding + classification)
- Publication entities are created by the approval flow
- PublicationWorker generates content via `IContentGenerator` and publishes

This ADR builds on top of that foundation. It does not contradict it; it extends the analysis pipeline (adding key facts extraction) and shifts the approval/publication granularity from Article to Event.

## Options

### Option 1 -- Key facts as part of existing ArticleAnalyzer

Add `KeyFacts` to `ArticleAnalysisResult` and have the existing `IArticleAnalyzer` (Gemini) extract them during the standard analysis phase. The same model call that produces category, tags, sentiment, summary also produces key facts.

**Pros:** Single AI call per article; no new interface needed; simpler pipeline.
**Cons:** Key facts extraction increases the prompt complexity and token usage of the primary analyzer model (Gemini Flash); if the analyzer model changes, key facts quality is coupled to it; the user explicitly requested a "cheap model" for facts, suggesting they want cost isolation.

### Option 2 -- Separate IKeyFactsExtractor interface with cheap model

Add a new `IKeyFactsExtractor` interface in `Core/Interfaces/AI/` that uses a dedicated cheap model (Haiku). ArticleAnalysisWorker calls it as an additional phase after enrichment, before event classification.

**Pros:** Cost-isolated (cheap model dedicated to facts); decoupled from main analyzer; follows SRP (each AI interface does one thing); user's explicit intent for cheap model is respected.
**Cons:** Additional AI call per article (latency); new interface and implementation to maintain.

### Option 3 -- Key facts extracted during event classification

Extract key facts only when an article is classified into an event (as part of `IEventClassifier`), since facts are primarily useful for event-level publication.

**Pros:** Facts are only generated when needed (article joins an event); saves cost for rejected/unclassified articles.
**Cons:** Contradicts user's request ("Article should gain a KeyFacts field"); classification already does significant work; facts would not be visible to the editor during article review.

## Decision

**Option 2 -- Separate IKeyFactsExtractor with cheap model.**

Rationale:
- The user explicitly stated "these key facts should be generated by a cheap model," indicating cost isolation is a requirement.
- The project already follows the pattern of one-purpose AI interfaces: `IArticleAnalyzer` (analysis), `IEventClassifier` (classification), `IEventSummaryUpdater` (summary updates), `IContentGenerator` (publication content). Adding `IKeyFactsExtractor` is consistent.
- Haiku is already configured in `AiOptions.Anthropic` (the `ClassifierModel` and `SummaryUpdaterModel` use Haiku). A new `KeyFactsExtractorModel` config field follows the same pattern.
- The additional latency is acceptable because key facts extraction can run in parallel with embedding generation (both happen after the analysis result is available).

### Detailed Design

#### 1. Domain Model Changes

**`Core/DomainModels/Article.cs`** -- add `KeyFacts` field:
```
public List<string> KeyFacts { get; set; } = [];
```
Placed in the "Ai" section alongside `Tags`, `Category`, `Sentiment`, `Summary`. This is a list of short factual statements (e.g., "Ukraine recaptured the village of X on April 3", "5 casualties reported").

**`Infrastructure/Persistence/Entity/ArticleEntity.cs`** -- add `KeyFacts` column:
```
public List<string> KeyFacts { get; set; } = [];
```
Stored as a PostgreSQL `jsonb` array, same pattern as `Tags` (which is already `List<string>` on the entity).

#### 2. New AI Interface

**`Core/Interfaces/AI/IKeyFactsExtractor.cs`**:
```
Task<List<string>> ExtractAsync(Article article, CancellationToken cancellationToken = default);
```
Input: an Article with at least `Title`, `Content`, and `Summary` populated (post-analysis).
Output: a list of 3-7 short factual statements.

**`Infrastructure/AI/HaikuKeyFactsExtractor.cs`** -- implementation using Anthropic Haiku. Follows the same constructor pattern as `ClaudeContentGenerator` (API key, model name from config).

#### 3. Configuration

**`Infrastructure/Configuration/AiOptions.cs`** -- add to `AnthropicOptions`:
```
public string KeyFactsExtractorModel { get; set; } = "claude-haiku-4-5-20251001";
```

#### 4. ArticleAnalysisWorker Pipeline Extension

The existing `ProcessArticleAsync` in `ArticleAnalysisWorker` currently runs:
1. Phase A: AI analysis (category, tags, sentiment, summary)
2. Phase B: Embedding generation + deduplication
3. Phase C: Event classification

This ADR inserts a new step between Phase A and Phase B:

1. Phase A: AI analysis (existing)
2. **Phase A2: Key facts extraction (new)** -- calls `IKeyFactsExtractor.ExtractAsync`, stores result in `Article.KeyFacts` via a new repository method `UpdateKeyFactsAsync`
3. Phase B: Embedding generation + deduplication (existing)
4. Phase C: Event classification (existing)

Phase A2 uses the article's `Content` and `Summary` (both available after Phase A). It can also run concurrently with Phase B (embedding generation) since they are independent, but sequential execution is acceptable for simplicity and follows the existing linear pattern in the worker.

The `AnalysisContext` record gains `IKeyFactsExtractor KeyFactsExtractor`.

#### 5. Event Approval Flow (replaces Article Approval)

**Current flow:** Editor approves individual Articles via `POST /articles/{id}/approve` -> `ArticleApprovalService.ApproveAsync` -> creates Publications linked to that Article.

**New flow:** Editor approves Events via `POST /events/{id}/approve` -> new `IEventApprovalService.ApproveAsync` -> creates Publications linked to the Event (not individual articles).

**`Core/Interfaces/Services/IEventApprovalService.cs`** (new):
```
Task<Event> ApproveAsync(Guid eventId, Guid editorId, List<Guid> publishTargetIds, CancellationToken cancellationToken = default);
Task<Event> RejectAsync(Guid eventId, Guid editorId, string reason, CancellationToken cancellationToken = default);
```

**`Infrastructure/Services/EventApprovalService.cs`** (new):
- `ApproveAsync`:
  1. Load Event with Articles via `IEventRepository.GetDetailAsync`
  2. Validate Event exists and is in `Active` status
  3. Validate all publish targets exist and are active
  4. Create one Publication per publish target, linked to the Event (via `Publication.EventId`)
  5. Update Event status to a new `EventStatus.Approved` value
  6. Update all Articles in the Event to `ArticleStatus.Approved`
  7. Return the Event

- `RejectAsync`:
  1. Load Event with Articles
  2. Update Event status to a new `EventStatus.Rejected` value
  3. Update all Articles in the Event to `ArticleStatus.Rejected`
  4. Return the Event

**`Core/DomainModels/Event.cs`** -- extend `EventStatus` enum:
```
public enum EventStatus
{
    Active,
    Approved,
    Rejected,
    Resolved,
    Archived
}
```

**`Core/DomainModels/Article.cs`** -- `ArticleStatus` keeps the existing values. Articles transition to `Approved`/`Rejected` as a batch when their Event is approved/rejected. The `AnalysisDone` status means "waiting for event-level approval."

#### 6. EventsController -- New Approval Endpoints

**`Api/Controllers/EventsController.cs`** -- add:

```
POST /events/{id}/approve   -- body: ApproveEventRequest { PublishTargetIds }
POST /events/{id}/reject    -- body: RejectEventRequest { Reason }
```

These follow the exact same pattern as the existing `ArticlesController.Approve` and `ArticlesController.Reject`.

**`Api/Models/EventDtos.cs`** -- add:
```
record ApproveEventRequest(List<Guid> PublishTargetIds);
record RejectEventRequest(string Reason);
```

#### 7. ArticlesController -- Remove Approval Endpoints

Remove `POST /articles/{id}/approve` and `POST /articles/{id}/reject` from `ArticlesController`. Articles are no longer approved individually; they are approved as part of their Event.

Remove `IArticleApprovalService` interface and `ArticleApprovalService` implementation (replaced by `IEventApprovalService`).

The `GET /articles` (pending for approval) endpoint changes semantics: it now lists articles in `AnalysisDone` status that have an EventId assigned (editor views them grouped by Event, not individually). This endpoint may be deprecated in favor of the Events list endpoint, but keeping it for now allows article-level browsing.

#### 8. Publication Model -- Event-Centric Content Generation

**`Core/DomainModels/Publication.cs`** -- the `Article` navigation property becomes optional (nullable). The primary link is now `EventId` (already exists on Publication). When a Publication is created for an Event approval, `EventId` is set and `ArticleId` is null (since the publication represents the entire Event, not one article).

However, this would be a significant entity restructuring. A simpler approach: keep `ArticleId` on Publication as a required FK but set it to the "Initiator" article of the Event (the first article). The `EventId` field (already present) links to the full Event. The `IContentGenerator` is updated to accept the Event context.

**Decision: Keep ArticleId as-is, use EventId for content generation context.**

- Publication is still created with `ArticleId` pointing to the Event's initiator article (for backward compatibility and Telegram threading).
- `Publication.EventId` is set to the Event being published.
- Content generation uses the Event, not just the Article.

#### 9. IContentGenerator -- Event-Aware Content Generation

**`Core/Interfaces/AI/IContentGenerator.cs`** -- change signature:

```
Task<string> GenerateForPlatformAsync(
    Event evt,
    PublishTarget target,
    CancellationToken cancellationToken = default,
    string? updateContext = null);
```

The method now receives the full Event (with its Articles loaded, including their Summaries and KeyFacts) instead of a single Article.

**`Infrastructure/AI/ClaudeContentGenerator.cs`** -- update the user prompt:

Instead of sending a single article's title/content/category/tags, send:
- Event.Title, Event.Summary
- For each Article in the Event: Article.Summary, Article.KeyFacts
- Combined tags and category from all articles

This produces a richer, multi-source publication that synthesizes information from all articles covering the same story.

#### 10. PublicationWorker -- Load Event Context

**`Worker/Workers/PublicationWorker.cs`** -- in `GenerateContentAsync`:

Currently fetches Publication with Article and PublishTarget. Must now also load the Event with its Articles (including their KeyFacts and Summaries).

**`Core/Interfaces/Repositories/IPublicationRepository.cs`** -- update `GetPendingForContentGenerationAsync` to include Event with Articles:
```
.Include(p => p.Event)
    .ThenInclude(e => e.Articles)
```

Or add a separate repository method to load the Event context when needed.

#### 11. IArticleRepository -- New Method

**`Core/Interfaces/Repositories/IArticleRepository.cs`** -- add:
```
Task UpdateKeyFactsAsync(Guid id, List<string> keyFacts, CancellationToken cancellationToken = default);
```

Uses `ExecuteUpdateAsync` pattern, storing key facts as a JSON array (same as Tags).

#### 12. Mappers

**Infrastructure/Persistence/Mappers/ArticleMapper.cs** -- add `KeyFacts` mapping in `ToDomain` and `ToEntity`.

**Api/Mappers/ArticleMapper.cs** -- add `KeyFacts` to `ArticleDetailDto` (so editors can see them).

**Api/Mappers/EventMapper.cs** -- add `KeyFacts` to `EventArticleDto` (so editors see key facts per article in the event detail view).

**Api/Models/ArticleDetailDto.cs** -- add `List<string> KeyFacts` parameter.

**Api/Models/EventDtos.cs** -- add `List<string> KeyFacts` to `EventArticleDto`.

#### 13. Database Migration

One EF Core migration:
- Add `KeyFacts` column to Articles table (jsonb, default empty array)
- No other schema changes (EventId on Publication already exists)

## Consequences

### Positive

1. **Richer publications** -- content generation draws from all articles covering an event, not just one article's perspective.
2. **Structured facts** -- key facts enable better event summaries, contradiction detection, and publication quality.
3. **Cost-efficient** -- key facts extraction uses Haiku (cheap), main analysis uses Gemini Flash (mid-tier), content generation uses Sonnet (quality).
4. **Editor workflow improvement** -- approving Events is more natural than approving individual articles; the editor sees the full picture of a story.
5. **Consistent with existing architecture** -- follows the one-purpose AI interface pattern, worker pipeline structure, and approval service pattern.

### Negative / Risks

1. **Additional AI call per article** -- key facts extraction adds latency and cost to the analysis pipeline (mitigated by using Haiku, the cheapest model).
2. **Breaking change to approval flow** -- removing article-level approval requires UI changes and may disrupt existing editorial workflows.
3. **IContentGenerator signature change** -- all implementations and callers must be updated. Currently only one implementation (`ClaudeContentGenerator`), so impact is contained.
4. **Publication query complexity** -- loading Event with all Articles for content generation increases query size.

## Implementation Notes

### For Feature-Planner

This change should be sequenced in three phases to keep the system working throughout:

**Phase 1 -- Key Facts (non-breaking addition):**
1. Add `KeyFacts` to Article domain model, entity, mappers, DTOs, migration
2. Add `IKeyFactsExtractor` interface and `HaikuKeyFactsExtractor` implementation
3. Add `UpdateKeyFactsAsync` to `IArticleRepository` and implementation
4. Extend `ArticleAnalysisWorker` to call key facts extraction after analysis
5. Deploy -- articles now have KeyFacts, everything else unchanged

**Phase 2 -- Event Approval (breaking change to approval flow):**
6. Add `Approved` and `Rejected` to `EventStatus` enum
7. Create `IEventApprovalService` interface and `EventApprovalService` implementation
8. Add approve/reject endpoints to `EventsController`
9. Remove approve/reject endpoints from `ArticlesController`
10. Delete `IArticleApprovalService` and `ArticleApprovalService`
11. Update DI registrations
12. Deploy -- editors now approve Events

**Phase 3 -- Event-Based Content Generation (breaking change to publication):**
13. Update `IContentGenerator` signature to accept `Event`
14. Update `ClaudeContentGenerator` to build event-based prompt
15. Update `PublicationWorker` to load Event context and pass to content generator
16. Update `IPublicationRepository` query to include Event with Articles
17. Deploy -- publications are generated from Event context
